# 中间塔防 - 开发计划

## 项目概述

**项目名称**：中间塔防（MidTowerDefense）
**项目类型**：局域网多人合作塔防游戏（1-4人）
**玩家分布**：中央水晶的上下左右四个方位
**核心玩法**：塔防 + 被动技能（肉鸽-like）

**单局流程**：
1. 选择初始塔（塔有不同攻击方式和被动技能）
2. 战斗阶段：抵御敌人波次
3. 阶段间：花费金币升级被动技能或重roll
4. 存活到单局结束

**被动技能**：9个技能槽位，初始0级，最高5级，升级时roll更高等级的技能
**套装效果**：同等级技能存在组合效果（如技能A+技能B触发套装效果）
**金币用途**：升级被动技能、重roll被动技能
**开发周期**：预计 3-4 个月

## 开发阶段

### 第一阶段：核心框架搭建

| 模块 | 主要任务 | 产出 |
|------|----------|------|
| 项目初始化 | 搭建开发环境、配置构建工具 | 可运行的基础项目 |
| 地图系统 | 俯视角地图、中央水晶、上下左右四个玩家方位 | 地图编辑器 |
| 实体系统 | 基础游戏实体（塔、敌人）的抽象 | Entity 基类 |
| 简单敌人AI | 敌人出生、移动、攻击基地 | 敌人波次系统 |

### 第二阶段：被动技能系统

| 模块 | 主要任务 | 产出 |
|------|----------|------|
| 初始塔选择 | 选择不同初始塔（攻击方式+被动） | 选塔界面 |
| 被动技能系统 | 9个技能槽位，0-5级升级，套装组合效果 | 被动技能逻辑 |
| 重roll系统 | 金币重roll当前等级被动 | 重roll UI |
| 防御塔攻击 | 塔的攻击方式和特效 | 3-4 种防御塔类型 |

### 第三阶段：多人联机系统

| 模块 | 主要任务 | 产出 |
|------|----------|------|
| 房间管理 | 创建/加入房间、玩家列表 | 房间 UI |
| 网络同步 | 状态同步、延迟处理 | 同步机制 |
| 同步验证 | 联机功能测试与优化 | 稳定的多人游戏 |

### 第四阶段：完善与优化

| 模块 | 主要任务 | 产出 |
|------|----------|------|
| 敌人多样化 | 更多敌人类型和特性，支持大量敌人同时存在 | 丰富敌人种类 |
| 技能池扩展 | 扩展被动技能池，增加更多技能类型 | 技能池丰富 |
| 被动技能升级 | 被动技能升级逻辑、金币消耗 | 升级平衡调整 |
| 界面优化 | 玩家界面、结算界面 | 完整 UI |
| 性能优化 | 大量敌人渲染、对象池、帧率保障 | 流畅游戏体验 |

---

## 里程碑

1. **M1 - 基础可玩版本**：单人模式，1 种防御塔，1 种敌人，可运行
2. **M2 - 核心功能版本**：单人模式，3+ 防御塔，5+ 敌人类型
3. **M3 - 联机版本**：支持 1-4 人局域网联机
4. **M4 - 发布版本**：功能完整，bug 修复，准备发布

## 技术栈

- **游戏框架**：原生 HTML5 Canvas + JavaScript
- **网络方案**：PeerJS (WebRTC P2P)
- **服务器**：无需后端

---

## 风险与备选方案

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| 大量敌人同时存在 | 性能下降、帧率降低 | 使用对象池、分批渲染、视口裁剪 |
| 多人同步复杂度高 | 开发延期 | 预留更多缓冲时间，优先验证同步方案 |
| 敌人AI设计复杂 | 体验不佳 | 先实现简单AI，后续迭代优化 |
| 资源创作周期长 | 项目阻塞 | 使用占位资源，优先保证功能开发 |
